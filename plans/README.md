## Findings considered and rejected

- **OTP email-param tampering** (`/login/check?email=` passed to `verifyOtp`):
  not a vulnerability. Supabase binds the OTP to the issuing email server-side; a
  token issued for email A will not verify against email B. By-design safe.
- **Middleware calls `auth.getUser()` on every request**: standard `@supabase/ssr`
  pattern, not a flaw. The matcher already excludes static assets. Skip unless
  production profiling shows real p95 latency.
- **`motion` library on all onboarding routes**: intentional UX polish, recently
  shipped on purpose. Not debt; leave it.
- **`packages/core` premature abstraction**: near-empty but harmless and cheap to
  keep; it'll earn its place when booking logic is shared web↔native. Revisit at
  that point, don't churn now.
- **`auth/confirm` token_hash length not validated**: Supabase validates the
  token server-side; a defensive length guard adds nothing actionable.
